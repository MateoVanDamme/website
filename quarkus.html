Problem description

This term project for my master combines everything I've learned so far about distributed applications and system design. This project implements a backend for a museum. Based on a functional analisys business scenarios are extracted. After an analysis phase the scenarios are converted into a distributed system architecture.
The application is developed completely in Java using Quarkus. The containers are deployed in a Kubernetes cluster that runs in the UGent datacenter.

Business scenarios

Ticket Sales: Users can purchase tickets valid for specific dates. Bulk ticket purchases are allowed. Capacity depends on the number of guards present.

Ratings and Recommendations: Users can view paintings online and give ratings. Curators use user feedback for museum management. Users can get personalized recommendations for future visits.

Artwork Catalog: There is a catalog that stores all paintings and exhibits. Paintings are assigned to spots in the museum and exhibits are planned.

Staff Management: Guard work schedules are maintained. Guards can have fixed or flexible contracts. Visitor capacity depends on the number of guards. Qualified guides are allocated for different exhibitions.

Technology

The application is developed completely in Java using Quarkus. The containers are deployed in a Kubernetes cluster that runs in the UGent datacenter. Data objects in our application are stored and managed in the MongoDB database using Quarkus Panache. Keycloak is employed as an identity and access management solution to secure and authenticate users in our application. In addition, SmallRye Mutiny was utilized to implement asynchronous and reactive programming.

Decomposition

The final system architecture splits the business scenarios into five contained microservices that each have their own tasks, privileges and endpoints. The service diagram below shows how the various services interact with each other.

Microservices
  • Tickets: Handles ticket and tour orders, with maximum capacity determined by the personnel service.
  • Personnel: Manages staff scheduling, including guards and exhibition guides, with a work schedule for each.
  • Catalog: Stores paintings and exhibitions. The curator can manage painting and exhibition data.
  • Ratings: Records visitor ratings for paintings. Ratings can be retrieved by curators.
  • Analytics: Generates personalized recommendations for visitors based on their ratings.
